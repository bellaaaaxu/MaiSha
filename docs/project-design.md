## 1. 产品定位

### 一句话定位
买啥是**一个人的买菜手账**：按店记下要买啥，到店不漏买；随手把清单发给家人，点开就能一起勾。为华人超市场景做到最深。

> 叙事金字塔（L1 个人价值 → L2 家庭放大 → L3 华人深度）与全部对外文案定稿见
> `docs/superpowers/specs/2026-07-02-unified-narrative-design.md`。写任何新文案前先过那里的「三问」。

### 谁会用
- **主要用户**：管买菜的人——一个人用就完整成立（想到就记、按店分好、到店不漏买）
- **放大场景**：家庭共用一张清单——一方记、一方买，链接点开即加入（零注册是核心增长机制）
- **首发滩头**：北美华人家庭（数据研判见 §9.3）；获客层显性打「华人超市」，产品层保持普适

### 不做谁 / 不做什么
- 通用待办/备忘录替代品（没有「按店」结构的记录，用 Notes 就够）
- 下单/配送履约（Weee!/T&T 线上是下游渠道不是对手，见 §9.4——买啥永远站在渠道上游的规划层）
- 食谱完整教学（下厨房、Paprika 已经做得很好）
- 营养卡路里追踪（Yazio、薄荷做得很好）
- 商业批量采购（B 端）

## 2. 设计哲学

### 2.1 三个不变性

1. **零注册、零登录**
   匿名 token 自动识别身份，链接打开即用。这是核心增长机制——把链接发微信，对方点开就加入，不需要任何账号操作。

2. **按超市组织，不按时间或分类**
   购物清单 app 的失败模式是"按购物日期"或"按分类"分。真实场景是：人在某个超市，只关心**这家店要买啥**。所以主视图按超市分组，分类只作为视觉锚点（emoji）。

3. **手感优先**
   PWA 必须像原生 app。安装到主屏、全屏沉浸、滑动顺滑、点击有反馈、动画轻量。每一次点击都要有视觉确认，比如添加成功时图标弹一下、勾选时 ✓ 渐入。

### 2.2 视觉语言

**温暖日系手绘风**。不是冷淡硅谷极简风。
- 主色调：暖米色（#faf6f0）渐变背景、墨绿（#7ca982）作为 accent、棕色系（#5a4e3c）作为文字
- 食材图标：**手绘水彩风**（铅笔线稿 + 柔和上色），像高级食谱书的插画
- 圆角：大胆使用（卡片 18px、按钮 12px、sheet 顶部 24px）
- 阴影：极轻，营造纸感而非玻璃感
- 动画：弹性自然，不要机械感

视觉规则总结在 `src/index.css` 的 CSS variables 和各组件的 inline style 里。颜色不随便加新的，必要时去 `feedback_ui_style.md`（用户偏好记忆）参考。

### 2.3 交互原则

**少打字、多图形选择**
中文输入麻烦，所以食材网格 + 常买区让用户基本只需要"点"。手输框作为兜底。

**反馈即时、操作可撤销**
- 添加物品：sheet 不自动关闭，可连续添加，已添加显示 ✓
- 点已添加的食材会**删除**（toggle 行为）而不是无效操作
- 完成采购前弹确认 modal

**共享但不强制同步焦点**
两个用户可以同时操作，互相能看到对方的添加和勾选，但不强制看到对方在哪屏（不做"实时光标"）。

## 3. 技术架构

### 3.1 技术栈

| 层 | 选择 | 原因 |
|---|---|---|
| **前端框架** | React 18 + TypeScript + Vite | 生态、HMR、PWA 插件成熟 |
| **样式** | Tailwind + inline style | Tailwind 处理 layout，复杂主题色用 inline style 更直接 |
| **路由** | react-router-dom v6 | 标准 |
| **状态** | React useState/useEffect + Supabase 订阅 | 单清单、低复杂度，不上 Redux/Zustand |
| **数据库** | Supabase (Postgres + Realtime) | 实时订阅 + RLS + 匿名认证，免运维 |
| **认证** | Supabase Anonymous Auth | 零注册体验的核心，token 存 localStorage |
| **拖拽** | @dnd-kit/core | 移动端触摸友好，touch-action 控制好 |
| **PWA** | vite-plugin-pwa | Workbox 集成、自动 manifest、SW 更新提示 |
| **部署** | Cloudflare Workers + Static Assets | 免费 CDN、GitHub 自动部署 |
| **测试** | Vitest + Testing Library | utils 单测覆盖核心逻辑 |

### 3.2 数据流

```
┌────────────────────────────────────────────────────┐
│  Browser (PWA)                                     │
│                                                    │
│  React Components                                  │
│       ↓                                            │
│  useAuth / useList / useItems / useRecipes 等 hook │
│       ↓                                            │
│  src/lib/db.ts (Supabase client wrapper)          │
│       ↓                                            │
│  @supabase/supabase-js                             │
└────────────────────┬───────────────────────────────┘
                     │
                  HTTPS + WebSocket
                     │
┌────────────────────▼───────────────────────────────┐
│  Supabase (Cloud)                                  │
│                                                    │
│  Postgres (lists, items, recipes, ...)             │
│       ↑                                            │
│  Row Level Security (auth.uid())                   │
│       ↑                                            │
│  Realtime (postgres_changes)                       │
│  Anonymous Auth                                    │
└────────────────────────────────────────────────────┘
```

**关键设计：**
- 单向数据流：Supabase 是 source of truth，前端订阅变化更新本地 state
- 离线兜底：service worker 缓存静态资源 + Supabase 调用走 NetworkFirst 短超时
- 实时更新：每个 list 的 items 表订阅 INSERT/UPDATE/DELETE 事件，1 秒内反映到 UI

## 5. 关键决策记录

### 5.1 为什么不用微信小程序

原本 2026-04-22 计划做微信小程序。**因为个人小程序注册要求管理员微信绑定银行卡**形成障碍，改用 React PWA + Supabase。意外收获是：PWA 跨设备（iOS/Android/桌面浏览器都能用）、迭代不需要审核、分享链接更通用。

### 5.2 为什么不用 Firebase

Supabase 的 Postgres + RLS 比 Firestore 的安全规则容易写，关系型数据更适合"items 关联 list"这种结构，Realtime 接口简单，且开源可自托管。

### 5.3 为什么不做完整账号系统

家庭场景"可信"——共享链接的人就是家人，不需要密码隔离。匿名 token 持久化即可。如果将来要做社区层（v3.1），届时再加可选的"绑定邮箱"功能保护社区身份。

### 5.4 为什么图标选水彩手绘风

主流购物 app 都用扁平 emoji 或线性图标，视觉同质化严重。水彩手绘风：
1. **情感温度**：让"买菜"这件日常琐事变得有质感
2. **品牌差异**：一眼能认出是"买啥"
3. **可扩展**：用 Gemini 按统一 prompt 生成，新食材几分钟搞定

### 5.5 为什么 AddSheet 不自动关闭

旧版每次添加一个 item，sheet 自动关闭。用户加 5 样东西要开关 5 次。改成"加完保持开启 + 已加显示 ✓"后，**用户能批量添加且看到自己加了啥**。配合"点已加的删除"形成 toggle 心智模型。

### 5.6 为什么部署到 Cloudflare 而不是 Vercel

最初用 Vercel。后来切到 Cloudflare 是因为：
1. Cloudflare 免费额度更宽松
2. Workers + Static Assets 原生 SPA 路由处理
3. 国内访问相对稳定（虽然依然有限）

## 6. 性能与限制

### 6.1 当前已知边界

- **单清单单成员上限**：Postgres array 字段无硬限制，但 UI 上 10+ 成员会拥挤
- **items 数量**：单个清单 100+ items 时滚动开始卡，Realtime 订阅压力增加
- **图标资产**：每个 ~30KB，60+ 图标 ≈ 2MB，service worker 全缓存，首次安装稍慢
- **Supabase 免费层**：超过 7 天无活动会暂停项目，需要重连

### 6.2 性能策略

- 图标使用 `<img>` 配 `loading="lazy"` + service worker 缓存
- AddSheet 的图标网格在打开时一次性渲染，没用虚拟列表（数量可控）
- Realtime 订阅按 list_id 过滤，不订阅整张表

## 7. 给协作者的速查

### 7.1 新功能怎么提

1. 在 GitHub Issues 写需求（或聊天里讨论）
2. 用 `/superpowers:brainstorming` 走完一遍设计澄清
3. 把 spec 文档写到 `docs/superpowers/specs/YYYY-MM-DD-<feature>-design.md`
4. 用 `/superpowers:writing-plans` 拆实施计划
5. 按计划实现，每个 commit 跟一个步骤

### 7.2 新图标怎么加

1. 决定食材名 + icon 文件名（kebab-case）
2. 去 `icon-prompts.md` 加一段 Gemini prompt
3. 用 Gemini 生成 PNG，存到 `public/icons/<name>.png`
4. 去 `src/utils/icon-registry.ts` 加一条 `{ name, icon, category, aliases? }`
5. 如果有同义词，作为 `aliases` 加进去（不要新建独立条目）

### 7.3 重要约定

- 注释只写**为什么**不写**是什么**
- 不写 TODO / 占位代码，缺什么补什么
- 文件超 300 行考虑拆分
- 颜色用 inline style 而不是新增 Tailwind class
- 不要乱加错误处理；只在系统边界（用户输入、API 返回）验证

## 8. 图标体系（生鲜写实 + 长尾纯色 + 梅兰竹菊集章）

> 2026-06-14 定案。解决"中超商品千千万，很多（零食/饮料/调味/美妆/医疗）无法用一张写实插画代表"与"水彩手绘是核心调性"之间的张力。
>
> **2026-07-05 修订**：长尾兜底从「食物小人班底（描绘）」换装为「梅兰竹菊装饰系 + 首字角标（装饰）」。小人是**描绘**，会错误声称「这件商品是茶叶蛋」——带来撞脸读作 bug、语义分配单向门、食物脸贴非食品违和三条结构性弊端；**装饰**不声称任何事，三题一起消解，识别功能由首字角标拿回。班底退役至品牌层/图鉴层，一张图不浪费。详见 [flora spec](superpowers/specs/2026-07-05-flora-decor-fallback-design.md)。
>
> **⚠️ 2026-07-08 再调整（决策已定，spec/实现待做，详见 §8.7）**：用户觉得梅兰竹菊当日常图标太花哨。① 长尾兜底改**纯色块**（不再用花）；② 梅兰竹菊 12 花转做**收藏印章「钤印集章」**（完成采购的奖励，非图标）；③ 277 预设图标转**日式淡雅**重制。下方 §8.1–§8.6 描述的是 07-05 的 flora 已上线状态，§8.7 是最新方向。

### 8.1 核心原则

品牌签名是**水彩这门语言**（手绘线稿、暖色配色、纸感），**不是**"每个 SKU 都有专属肖像"。一旦把两者分开，问题即化解：**所有东西都说水彩话，而每个图标的"具体程度"按它好不好画优雅降级。**

### 8.2 三层降级

- **Tier 1 · 生鲜整食**（肉/菜/果/海鲜）→ 专属写实水彩插画（现有 276 预设的延续）。**王牌，既萌又能认，永远没脸。** 值得持续投入。
- **Tier 2 / 3 · 其余一切**（零食/饮料/调味/美妆/医疗/日用）→ **梅兰竹菊装饰系（12 只无脸雅集花卉）+ 首字角标**（2026-07-05 换装），按商品名 rendezvous 稳定分配；**名字与角标负责识别，花负责治愈**。装饰框架下撞图=贴纸复用（非 bug）、无脸与 Tier 1 同语言、公有领域纹样零版权风险。
- **备选**：花图加载失败回退**水彩文字色块**（WatercolorFallback，最后防线）。

### 8.3 吉祥物：小榕包 + 食物小人班底

- **队长（主吉祥物）**：大名「**小榕包**」，本体是一只**小笼包**（精致小巧，刻意区别于北方大包子）。名字三关：藏创始人名字"榕" + 谐音"小笼包" + 与班底成员撞名眨眼。由来见 §8.6 / 记忆。
  - 职责：onboarding、空状态、加载、完成庆祝、收藏品图鉴 C 位、偶尔头部卖萌、app 图标点睛。
  - **定妆（canonical，2026-06-14 首版出图确认）**：圆豆眼带高光、**无睫毛**（保持中性 + 小尺寸清晰 + 与班底统一）；**头两侧各一只小圆耳朵 = 队长专属签名**（班底不一定有）；圆肉脸 + 腮红 + 褶子收口，暖米纸感水彩 + 铅笔线稿。canonical 参考已入库：`docs/brand/xiaorongbao-canonical.webp`（1024² q90，2026-07-04）——所有后续班底生成/收藏品/app 图标以此对脸；staging 的 2048² 原图建议用户另行归档（网盘/相册），5 只探索稿班底（不合定妆铁律）可删或留作社媒过程素材。
- **班底（约 30 只，"中国各地特色美食"图鉴）**——**2026-07-05 起退役至品牌层/图鉴层**（onboarding/空状态/完成庆祝/收藏品专职，不再任商品图标兜底；已出图 10 只定稿存 `mascot-staging/final/`）。铁律不变：**只用"做好的料理/点心"（绝不碰生鲜写实，避免与 Tier 1 撞画风）；永远有脸。**
  - 北方/京津冀：大包子、饺子、馒头、糖葫芦、煎饼果子、驴打滚、茶叶蛋
  - 东北：黏豆包、烤地瓜
  - 江浙沪/江南：生煎、青团\*、汤圆\*、粽子\*、麻薯团子
  - 广东/港式：虾饺、烧麦、叉烧包、菠萝包、蛋挞、鸡蛋仔
  - 川渝：红油抄手、担担面、糖油果子
  - 云贵：过桥米线
  - 西北/新疆：肉夹馍、馕、羊肉串
  - 闽台：珍珠奶茶、凤梨酥
  - 节令甜点/饮品：月饼\*、盖碗茶、豆花
  - 标 \* = 季节限定（青团清明 / 汤圆元宵 / 粽子端午 / 月饼中秋），可作收藏品图鉴的"限定款"解锁。

### 8.4 ¥0 引擎：有界装饰池 + 稳定分配

- 长尾图标**不逐件生成**——`rendezvous(归一化商品名, 成员id)`，**同一件商品永远同一张贴纸**（稳定不跳、扩池不洗牌——成员 id 永不改名是单向门）。2026-07-05 起池子 = 梅兰竹菊装饰系 12 只等权重（队长加权随小人退役取消）。
- 装饰池是**一次性资产**（12 花经 sharp + WebP 压缩入 precache ≈ 120KB），无限 SKU 零边际成本。
- **AI 运行时生成只留给真正值得画的生鲜新品（Tier 1）**——省配额、省成本。
- 工程教训（2026-07-05）：FNV-1a 需加 fmix32 终混（虚拟节点雪崩不足会让加权失效）；`loading="lazy"` 与预设图标行为不一致，不用。

### 8.5 出图：Claude 设计、Gemini 作画

> Claude（写代码/写字的模型）在对话里只能产出**矢量占位替身**（SVG/HTML），**不能画水彩**；水彩是 Gemini 图像模型的活（即现有图标管线）。分工：Claude 写 prompt + 定形/性格/布局，**真图拿 prompt 去 Gemini app 生成**。

**队长 prompt**（套用 `icon-prompts.md` 同一风格块，保证同一只手）：

```
生成一个手绘素描+柔和上色风格的吉祥物图标：

角色：一只圆滚滚的卡通小笼包（买啥吉祥物「小榕包」），白白胖胖、精致小巧，
顶部有自然的小笼包褶子收口；头两侧各一只小圆耳朵（队长专属签名）；
圆圆的肉脸、一对带高光的小圆豆眼（无睫毛）、红扑扑的腮红、温柔的微笑；
治愈、呆萌、招人疼
风格要求：
- 铅笔线稿描边，线条自然有手绘感
- 水彩/彩铅柔和上色，保留笔触和晕染感
- 像手账日记 / 绘本里的可爱角色插画
- 暖米色或纯白背景，角色居中，占画面 70-80%
- 256×256 正方形，无文字无装饰无阴影
- 不要任何文字和 logo
```

**班底模板**：**风格要求块一字不改**（§8.5 队长 prompt 里那段），只换「角色」行——并把下面这句**定妆约束原样抄进角色行**，保证 32 只是一家人：

```
角色：一只卡通{菜名}，{一句剪影/颜色特征，如「红亮糖球串在竹签上」}。
定妆：圆豆眼 + 单高光、一道弧形微笑、两坨腮红；只有食物本体和脸，
没有手脚四肢、不拿任何道具、没有鼻子、没有耳朵。
```

- **耳朵 / 手脚 / 姿势 / 道具 = 队长小榕包专属**；班底一律不要（小图才清晰、全家才统一）。
- **剪影 + 颜色要彼此分得开**——判定标准：「缩到 44px、放进同一列表，一眼能区分」。撞脸的（汤圆↔麻薯、大包子↔队长）择一或改色 / 加容器区分。

**梅兰竹菊装饰系模板**（2026-07-05 增，商品图标层现役）：风格要求块仍一字不改，内容行 + 无脸约束句：

```
生成一个手绘素描+柔和上色风格的装饰图标：

内容：{一句剪影/颜色特征，如「深褐枝干斜出，缀粉红五瓣梅花和花苞」}。
约束：纯植物，没有脸、没有眼睛、没有表情、不拟人；
构图简洁、剪影清晰，缩小到很小尺寸仍一眼可辨。
```

- **撞形判定**（替代撞脸判定）：圆瓣团型（菊/牡丹/山茶/芙蓉）至多取一；枝+粉花型至多取一（梅在役，海棠/桃花不进）。花名册 12 只与角色行定稿见 [flora spec](superpowers/specs/2026-07-05-flora-decor-fallback-design.md) §1。

### 8.6 为什么这条路成立

延续 §5.4（水彩的情感温度 + 品牌差异 + 可扩展），补上**可扩展性的真解**：可爱 ≠ 昂贵，只要"有界班底 + 稳定分配"。食物小人比通用动物更进一步——**是买啥独有、可被认领的品牌资产**，且天然能喂"收藏品图鉴"（见 ROADMAP 路线图）。队长名「小榕包」源自创始人小时候外号"包子（圆圆肉脸）"+ 名字"榕"，是抄不走的品牌根。

**③ 277 预设图标转「日式淡雅」重制** — 现有 277 预设从「黑线 + 饱和水彩」重制为「日式手绘绘本」气质：**淡褐色细线（非黑硬线）+ 低饱和薄涂 + 多留白 + 和风食谱插画感**。prompt 方向已认可（番茄/鸡蛋试样对味），待批量重制全 277。出图脚本要点：日式绘本 prompt + 每变体不同构图 + **md5 去重**（防 API 返回重复图）+ 请求间隔 sleep。

## 9. 市场与竞争研判（2026-06 数据调研）

> 2026-06-14 用 deep-research 跑了一轮带引用的竞品/市场调研（100 条事实抽 25 条做对抗式验证，19 存活）。以下是结论，判断性外推已标注。

### 9.1 品类现实：红海 + 高流失

- 共享购物清单是成熟红海，被免费/freemium 占满（AnyList 4.9★/约 7.9 万评分；JOJO 即时链接分享；Listonic 靠 FMCG 广告变现）。**正面拼通用清单会输给 Bring!/AnyList 和备忘录。**
- **高流失是品类常态**：全行业 D1 约 25.3% / D30 约 5.7%（Statista/AppsFlyer 互证），工具类 D30 低至约 2.44%。**留存是生死线——功能回答不了，只有真实用户能。**

### 9.2 唯一护城河：中餐 + 华人 niche

- 西方竞品**不服务中餐/华人**（AnyList 仅英/德，无中文无中餐食材）。
- App Store 仅有一个**已弃坑**的"购物清单·海外华人"（2022 停更、1 评分、无分享/微信/协作）——**位子有人占、但需求未被验证也未被满足**（弱推断，勿当"需求已证"）。
- 结论：凡**加深"中餐家庭 + 跨华人超市 + 微信零摩擦分享 + 水彩中餐图标"**这个 niche 的 = 战略级；通用清单功能 = 入场券。

### 9.3 滩头：北美华人 > 中国大陆（数据支持）

- 美国华人 550 万（2023，集中纽约/洛杉矶/旧金山/加州），加拿大华人 170 万（2021，多伦多+温哥华近 70%；列治文 54.3%、万锦 47.9%）——**规模真实、地理高度集中**（微信裂变 + 线下口碑可高效铺开）。
- 大陆面对美团等超级 app 深度吃占买菜（2024 营收 3376 亿、本地商业 2502 亿、7.7 亿交易用户）+ 安卓主导分发——**竞争与触达成本显著更高**。
- **招牌"查超市"的数据源在北美可用**：Apple Maps 对 99 Ranch 等有结构化「Asian Grocery Store」POI（地址/营业时间/Yelp 评分，多点核实）。

### 9.4 定位补充：规划层 ≠ 履约层

- Weee!（北美最大亚洲杂货电商，2024 营收 >10 亿美元、约 150 万买家）、大温饭团、T&T 网站、生鲜配送、UberEats 等都是**履约渠道（怎么买到）**，站在买啥**下游**，**不是正面竞品**。
- 它们的间接威胁是**蚕食"线下那一趟"**（每转一单线上 = 少一次线下清单）。但买啥的甜区——**跨多店 + 挑生鲜的中餐采购**——恰是配送的弱项，配送抢不走。
- **买啥 = 规划工具（想买什么），定位在所有渠道的上游。** v1 专注把"线下那趟"做到极致；远期可"渠道中立"。**不做下单/配送**（去 Weee! 主场送人头）。

### 9.5 对 v1 的含义

1. **别再打磨已建好的核心**——精简上架。
2. **补留存埋点**（品类高流失，先量再优化）。
3. **优先验证微信分享假设**（核心增长地基，本轮无量化来源——最便宜也最关键的验证）。
4. **查超市**：用户定为 v1 招牌（2026-06-14）。节奏建议——排作首发线最后一棒（唯一原生 Swift + 定位权限、最不确定），若临门卡住降级为 v1.1，不让上架日被最高风险代码绑架。

### 9.6 三个「未闭环」的核查结果（2026-06-14 聚焦调研）

- **#1 微信假设 → 部分闭环（分代际），且风险比想象的低。** 一代华人移民里微信占主导（纽约老年移民 2018：71% 用微信 vs 4% WhatsApp；卡尔加里一代 2020：微信高频）——核心「管采购的妈妈/婆婆」多属此列，假设有旁证。但**二代/在地出生华裔偏 iMessage/SMS**（18–35 岁东亚裔 2021：仅 27.6% 用微信联系家人）；无全人口比例数据。**关键反转**：买啥的分享本就是「发一条链接」，**渠道无关**（微信/iMessage/SMS 都能点开）——增长引擎不依赖「微信」这个平台，只依赖「人会把链接分享给家人」这个普适行为。**仍需用户验证的是行为（是否真用手机协调买菜），不是平台。**
- **#2 高德数据 → 完全闭环（是）；NA POI → 无面上审计，但两个新发现。** Apple 官方法律页确认：大陆 Apple Maps 自 iOS 6（2012）起用高德（Amap/AutoNavi）。✅ 彻底闭环。NA 侧：① **旧担忧被推翻**——「Apple Maps 北美 POI 主要来自 Yelp / Yelp 库仅 Google 三分之一」两条均被验证否决，北美覆盖不该用 Yelp 时代的悲观假设衡量，大概率比想的好。② **新增技术风险**：`MKLocalSearch`（store-finder 所用 API）有**漏返**的已知问题——Maps app 里可见的店，API 可能返回 0（Apple 承认）。store-finder 真机冒烟须专测「API 返回 vs Maps app 实见」的差距。
- **#3 清单类留存基准 → 确认不存在，但拿到更贴的代理。** 无 shopping-list/to-do 专属公开 benchmark。但**「生产力/清单类」比「工具类/电商」更贴**：UXCam 生产力类 D1 40–50% / **D30 12–18%**（vs 电商 D30 3–6%、工具类下限约 2.4%）。规划锚点上修——把买啥当**复用型清单工具（D30 约 12–18% 量级）**，别用 2.4% 那个最坏下限吓自己；最终以**自己埋点**为准。

> 来源：Apple 官方法律页（高德）、JMIR Aging / IJoC / Sage 等学术调查（微信分代际）、UXCam/BusinessofApps（留存代理）。仍开放：2022 后的分代际微信调查、NA 亚洲超市 POI 面上审计。

**主要来源**：Pew Research（美国华人）、StatCan 2021 普查（加拿大华人）、美团 2024 年报、Apple Maps POI 实测、BusinessofApps/AppsFlyer（留存）、App Store 各竞品页。
